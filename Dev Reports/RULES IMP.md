# Developer Report Guidelines

All development reports must follow this standardized structure.

Reports are meant to document meaningful progress only.
Avoid logging minor changes or micro-fixes.

Each report should represent a logical milestone or completed task set.

---

## File Naming Convention

devlog_XX.md

Example:
devlog_01.md
devlog_02.md

---

## Report Structure

### 1. Header

Devlog Number  
Date  
Developer Name  

---

### 2. Summary

A concise overview of what was accomplished.

---

### 3. Work Completed

Bullet-point list of completed tasks and implemented features.

Focus on:
- Architecture changes
- Feature additions
- Refactors
- Major fixes

Avoid documenting trivial edits.

---

### 4. Technical Notes

Optional section.

Include:
- Design decisions
- Architecture updates
- Important logic changes
- Known limitations

---

### 5. Next Steps

List of planned tasks or future improvements.

---

## Writing Rules

- Use formal technical language.
- Be clear and precise.
- No casual tone.
- No unnecessary detail.
- Each report must include developer name.
- Reports should only be created for meaningful progress.

---

## Example Header

Devlog 01  
Date: February 9, 2026  
Developer: Yazdan  

---

This structure must be followed by all contributors
